# SQL Notes Part 2 — Updating Data & The Company Database

## 1. Updating Records — `UPDATE`

Modify existing rows. **Always pair with `WHERE`**, or every row in the table gets updated.

```sql
-- Update a single column
UPDATE student
SET major = 'Bio'
WHERE major = 'Biology';

-- Update multiple columns at once
UPDATE student
SET name = 'Tom', major = 'undecided'
WHERE student_id = 1;
```

## 2. Deleting Records — `DELETE`

```sql
-- Delete ALL rows in the table (table itself still exists)
DELETE FROM student;

-- Delete only matching rows
DELETE FROM student
WHERE student_id = 5;
```

> ⚠️ `DELETE FROM table;` with no `WHERE` wipes every row — use carefully.

## 3. Constraint Behavior on Insert

| Violation | Result |
|---|---|
| Missing required (`NOT NULL`) column | Insert fails |
| Duplicate value in a `UNIQUE` column | Insert fails ("not working") |
| Valid, non-duplicate values | Insert succeeds |

```sql
INSERT INTO student VALUES (5, 'omar sultan', 'SWE');           -- works
INSERT INTO student VALUES ('omar faruk vai', 'EEE');           -- fails: missing id
INSERT INTO student (major) VALUES ('EEE');                      -- fails: name is NOT NULL
```

## 4. Advanced `SELECT`

**Sort by multiple columns** (second column breaks ties in the first):
```sql
SELECT *
FROM student
ORDER BY major, student_id;
```

**Limit number of rows returned:**
```sql
SELECT * FROM student
ORDER BY student_id DESC
LIMIT 2;
```

**Match against a list of values — `IN`:**
```sql
SELECT *
FROM student
WHERE name IN ('Claire', 'Kate', 'Mike');
```

**Comparison operators recap:**

| Operator | Meaning |
|---|---|
| `=` | equals |
| `<>` | not equals |
| `>` | greater than |
| `<` | less than |
| `>=` | greater than or equal |
| `<=` | less than or equal |

## 5. The Company Database

A realistic multi-table schema demonstrating **primary keys**, **foreign keys**, **composite keys**, and **referential actions**.

### Entity Overview

| Table | Primary Key | Foreign Key(s) |
|---|---|---|
| `employee` | `emp_id` | `super_id` → `employee(emp_id)`, `branch_id` → `branch(branch_id)` |
| `branch` | `branch_id` | `mgr_id` → `employee(emp_id)` |
| `client` | `client_id` | `branch_id` → `branch(branch_id)` |
| `works_with` | composite (`emp_id`, `client_id`) | `emp_id` → `employee`, `client_id` → `client` |
| `branch_supplier` | composite (`branch_id`, `supplier_name`) | `branch_id` → `branch(branch_id)` |

**Color legend:** 🔴 Primary Key · 🟢 Foreign Key · 🔵 Attribute

### Full Schema (DDL)

```sql
CREATE DATABASE company_database;
USE company_database;

CREATE TABLE employee (
    emp_id      INT PRIMARY KEY,
    first_name  VARCHAR(40),
    last_name   VARCHAR(40),
    birth_date  DATE,
    sex         VARCHAR(1),
    salary      INT,
    super_id    INT,
    branch_id   INT
);

CREATE TABLE branch (
    branch_id       INT PRIMARY KEY,
    branch_name     VARCHAR(40),
    mgr_id          INT,
    mgr_start_date  DATE,
    FOREIGN KEY (mgr_id) REFERENCES employee(emp_id) ON DELETE SET NULL
);

-- Added after both tables exist, to avoid a circular dependency at creation time
ALTER TABLE employee
ADD FOREIGN KEY (super_id) REFERENCES employee(emp_id) ON DELETE SET NULL;

ALTER TABLE employee
ADD FOREIGN KEY (branch_id) REFERENCES branch(branch_id) ON DELETE SET NULL;

CREATE TABLE client (
    client_id   INT PRIMARY KEY,
    client_name VARCHAR(40),
    branch_id   INT,
    FOREIGN KEY (branch_id) REFERENCES branch(branch_id) ON DELETE SET NULL
);

CREATE TABLE works_with (
    emp_id       INT,
    client_id    INT,
    total_sales  INT,
    PRIMARY KEY (emp_id, client_id),
    FOREIGN KEY (emp_id)    REFERENCES employee(emp_id) ON DELETE CASCADE,
    FOREIGN KEY (client_id) REFERENCES client(client_id) ON DELETE CASCADE
);

CREATE TABLE branch_supplier (
    branch_id      INT,
    supplier_name  VARCHAR(40),
    supply_type    VARCHAR(40),
    PRIMARY KEY (branch_id, supplier_name),
    FOREIGN KEY (branch_id) REFERENCES branch(branch_id) ON DELETE CASCADE
);
```

### Why `employee` ↔ `branch` is split with `ALTER TABLE`
`employee.branch_id` references `branch`, but `branch.mgr_id` references `employee` — a **circular dependency**. The fix: create `employee` first (without the `branch_id` FK active yet), then create `branch`, then `ALTER TABLE employee` to add the foreign keys afterward.

### Why `works_with` & `branch_supplier` need composite keys
- In `works_with`, neither `emp_id` nor `client_id` alone uniquely identifies a row (an employee can have many clients, a client can have many employees) — together they do.
- In `branch_supplier`, neither `branch_id` nor `supplier_name` alone is unique (a branch can have many suppliers, a supplier can supply many branches) — together they form the composite primary key.
- Notably, **every column in these composite keys is also a foreign key**.

### `ON DELETE` Referential Actions

| Action | Effect when the referenced row is deleted |
|---|---|
| `ON DELETE SET NULL` | The foreign key column is set to `NULL` (used for `mgr_id`, `super_id`, `branch_id` — optional relationships) |
| `ON DELETE CASCADE` | The dependent row is **deleted too** (used for `works_with`, `branch_supplier` — rows that don't make sense without the parent) |

## Quick Reference Summary

- **`UPDATE ... SET ... WHERE ...`** — modify rows (omit `WHERE` → affects all rows)
- **`DELETE FROM ... WHERE ...`** — remove rows (omit `WHERE` → deletes all rows)
- **`ORDER BY col1, col2`** — sorts by col1, then breaks ties with col2
- **`WHERE col IN (...)`** — match against a list of values
- **Composite key** = primary key spanning 2+ columns, each often also a foreign key
- **`ON DELETE SET NULL`** — for optional relationships; **`ON DELETE CASCADE`** — for dependent rows
- Circular FK dependencies (e.g. employee ↔ branch) are resolved with `ALTER TABLE ... ADD FOREIGN KEY` after both tables exist
