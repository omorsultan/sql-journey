# SQL Notes Part 3 ‚Äî ON DELETE, Aggregates, Wildcards, UNION & JOINs

## 1. `ON DELETE` ‚Äî Foreign Key Behavior

What happens to **child rows** (in `orders`) when a **parent row** (in `users`) is deleted.

### `ON DELETE CASCADE`
```sql
FOREIGN KEY (user_id)
REFERENCES users(id)
ON DELETE CASCADE;
```
```sql
DELETE FROM users WHERE id = 1;
```

**users ‚Äî after delete**
| id | name |
|---|---|
| ~~1~~ | ~~Omar~~ *(deleted)* |
| 2 | Alice |

**orders ‚Äî after delete**
| id | user_id | product |
|---|---|---|
| ~~101~~ | ~~1~~ | ~~Laptop~~ *(deleted)* |
| ~~102~~ | ~~1~~ | ~~Mouse~~ *(deleted)* |
| 103 | 2 | Phone |

‚û°Ô∏è Orders 101 and 102 are **automatically deleted** along with their parent.

### `ON DELETE SET NULL`
```sql
FOREIGN KEY (user_id)
REFERENCES users(id)
ON DELETE SET NULL;
```
```sql
DELETE FROM users WHERE id = 1;
```

**users ‚Äî after delete**
| id | name |
|---|---|
| ~~1~~ | ~~Omar~~ *(deleted)* |
| 2 | Alice |

**orders ‚Äî after delete**
| id | user_id | product |
|---|---|---|
| 101 | `NULL` *(set null)* | Laptop |
| 102 | `NULL` *(set null)* | Mouse |
| 103 | 2 | Phone |

‚û°Ô∏è Orders 101 and 102 **survive**, but lose their link to the deleted user.

### Comparison

| Feature | ON DELETE CASCADE | ON DELETE SET NULL |
|---|---|---|
| Child rows | Deleted | Kept |
| Foreign key | Removed with row | Set to `NULL` |
| Nullable FK required? | No | Yes |
| Best for | Order items, comments | Posts, logs, documents |

- **CASCADE** üóëÔ∏è ‚Äî delete *everything* related; the child has no meaning alone.
- **SET NULL** üîó ‚Äî keep the child, remove the relationship; the data still matters.

## 2. Inserting Around a Circular Foreign-Key Relationship

`employee.branch_id` ‚Üí `branch`, and `branch.mgr_id` ‚Üí `employee` form a circular dependency, so data must be inserted in stages and patched with `UPDATE`:

```sql
-- Corporate
INSERT INTO employee VALUES(100, 'David', 'Wallace', '1967-11-17', 'M', 250000, NULL, NULL);

INSERT INTO branch VALUES(1, 'Corporate', 100, '2006-02-09');

UPDATE employee
SET branch_id = 1
WHERE emp_id = 100;

INSERT INTO employee VALUES(101, 'Jan', 'Levinson', '1961-05-11', 'F', 110000, 100, 1);
```

This same `INSERT ‚Üí UPDATE` pattern repeats for each branch's manager, since the manager's `branch_id` can't be set until the branch row exists, and the branch's `mgr_id` can't reference an employee until that employee exists.

## 3. Column Aliases ‚Äî `AS`

Renames a column in the result set without changing the underlying table.

```sql
-- Find the forename and surnames of all employees
SELECT first_name AS forename, last_name AS surname
FROM employee;
```
‚û°Ô∏è Results are labeled `forename` and `surname` instead of `first_name` / `last_name`.

## 4. Aggregate Functions

| Function | Purpose |
|---|---|
| `COUNT()` | Counts rows |
| `AVG()` | Average value |
| `SUM()` | Total value |
| `GROUP BY` | Groups rows sharing a value, for use with aggregates |

```sql
-- Find out all the different genders
SELECT DISTINCT sex FROM employee;          -- e.g. M, F

-- Find the number of employees
SELECT COUNT(emp_id)
FROM employee;

-- Find the average of all employee salaries
SELECT AVG(salary)
FROM employee;

-- Find the average salary of male employees only
SELECT AVG(salary)
FROM employee
WHERE sex = 'M';

-- Find the sum of all employee salaries
SELECT SUM(salary)
FROM employee;

-- Find out how many males and females there are
SELECT COUNT(sex), sex
FROM employee
GROUP BY sex;

-- Find the total sales of each salesman
SELECT SUM(total_sales), emp_id
FROM works_with
GROUP BY emp_id;
```

**Result ‚Äî total sales per employee:**

| SUM(total_sales) | emp_id |
|---|---|
| 282,000 | 102 |
| 218,000 | 105 |
| 31,000 | 107 |
| 34,500 | 108 |

‚û°Ô∏è Tells us how much each employee has sold in total.

## 5. Wildcards ‚Äî `LIKE`

Used with `LIKE` for pattern matching in `WHERE` clauses.

| Symbol | Meaning |
|---|---|
| `%` | Any number of characters (including zero) |
| `_` | Exactly one character |

```sql
-- Find any clients who are an LLC
SELECT *
FROM client
WHERE client_name LIKE '%LLC';
```
‚û°Ô∏è The `%` means any number of characters can come before `LLC` ‚Äî so it catches every company name ending in "LLC".

```sql
-- Find any branch suppliers who are in the label business
SELECT *
FROM branch_supplier
WHERE supply_type LIKE '%Label%';
```

## 6. `UNION`

Combines the results of **multiple `SELECT` statements** into a single result set/column.

**Requirements:**
- Each `SELECT` must return the **same number of columns**.
- Columns being combined must have **similar/compatible data types**.

```sql
-- Find a list of employee and branch names
SELECT first_name
FROM employee
UNION
SELECT branch_name
FROM branch;
```
‚û°Ô∏è Returns employee first names *and* branch names together, in a single column.

```sql
-- Find a list of all clients & branch suppliers' names
SELECT client_name
FROM client
UNION
SELECT supplier_name
FROM branch_supplier;
```

```sql
-- Same, but also include the related branch_id
SELECT client_name, client.branch_id
FROM client
UNION
SELECT supplier_name, branch_supplier.branch_id
FROM branch_supplier;
```

## 7. `JOIN`s

A **JOIN** combines rows from two or more tables based on a **related (shared) column** between them.

**Setup example** ‚Äî adding a branch with no manager yet:
```sql
INSERT INTO branch VALUES(4, 'Buffalo', NULL, NULL);
```

| branch_id | branch_name | mgr_id | mgr_start_date |
|---|---|---|---|
| 1 | Corporate | 100 | 2006-02-09 |
| 2 | Scranton | 102 | 1992-04-06 |
| 3 | Stamford | 106 | 1998-02-13 |
| 4 | Buffalo | NULL | NULL |

The **shared column** between `employee` and `branch` is the employee ID:
- `branch.mgr_id` stores an `employee.emp_id`.

### (Inner) `JOIN`
Only includes rows where the related column **matches in both tables**.

```sql
-- Find all branches and the names of their managers
SELECT employee.emp_id, employee.first_name, branch.branch_name
FROM employee
JOIN branch
ON employee.emp_id = branch.mgr_id;
```

**Result (3 rows)** ‚Äî only employees who are actual branch managers appear:
| emp_id | first_name | branch_name |
|---|---|---|
| 100 | David | Corporate |
| 102 | Michael | Scranton |
| 106 | Josh | Stamford |

### `LEFT JOIN`
Includes **all rows from the left table** (`employee`), even if there's no match in the right table (`branch`). Unmatched columns are filled with `NULL`.

```sql
SELECT employee.emp_id, employee.first_name, branch.branch_name
FROM employee
LEFT JOIN branch
ON employee.emp_id = branch.mgr_id;
```

**Result (9 rows)** ‚Äî every employee appears, not just managers:
| emp_id | first_name | branch_name |
|---|---|---|
| 100 | David | Corporate |
| 102 | Michael | Scranton |
| 106 | Josh | Stamford |
| 101 | Jan | NULL |
| 103 | Angela | NULL |
| 104 | Kelly | NULL |
| 105 | Stanley | NULL |
| 107 | Andy | NULL |
| 108 | Jim | NULL |

### `RIGHT JOIN`
Includes **all rows from the right table** (`branch`), even if there's no match in the left table (`employee`).

```sql
SELECT employee.emp_id, employee.first_name, branch.branch_name
FROM employee
RIGHT JOIN branch
ON employee.emp_id = branch.mgr_id;
```

**Result (4 rows)** ‚Äî every branch appears, including the one with no manager:
| emp_id | first_name | branch_name |
|---|---|---|
| 100 | David | Corporate |
| 102 | Michael | Scranton |
| 106 | Josh | Stamford |
| NULL | NULL | Buffalo |

### Comparing the JOIN types

| JOIN type | Rows included |
|---|---|
| `JOIN` (inner) | Only rows with a match in **both** tables |
| `LEFT JOIN` | **All** rows from the left table + matches from the right (`NULL` if none) |
| `RIGHT JOIN` | **All** rows from the right table + matches from the left (`NULL` if none) |
| `FULL JOIN`* | All rows from **both** tables, matched where possible, `NULL` where not |

*There's a fourth JOIN type, `FULL JOIN` (a.k.a. `FULL OUTER JOIN`), combining the behavior of `LEFT` and `RIGHT JOIN` ‚Äî it returns every row from both tables.

## Quick Reference Summary

- **`ON DELETE CASCADE`** ‚Üí deletes dependent child rows; **`ON DELETE SET NULL`** ‚Üí keeps child rows, nulls the FK
- Circular FK relationships are resolved with staged `INSERT` + `UPDATE` statements
- **`AS`** renames a column in the output only
- **Aggregates**: `COUNT()`, `AVG()`, `SUM()` ‚Äî pair with `GROUP BY` to aggregate per category
- **Wildcards**: `%` = any number of characters, `_` = exactly one character, used with `LIKE`
- **`UNION`** stacks results from multiple `SELECT`s into one column set (same column count & compatible types required)
- **JOINs**: `JOIN` (matches only) ¬∑ `LEFT JOIN` (all of left) ¬∑ `RIGHT JOIN` (all of right) ¬∑ `FULL JOIN` (all of both)
